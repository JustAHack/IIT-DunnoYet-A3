Testing

How will you test your project?

Testing products has a wide variety of different approaches and techniques, some overlapping. Testing methods include user testing and software development processes such as system tests including unit testing and integration tests. TiMe will incorporate system testing during the development and maintenance periods. This will include testing the business logic such as how shifts are created, updated, and deleted, how users are created, updated, and deleted etc. User testing is important to our project as our product should be built for our target audience and fit their needs, not the needs we believe they have. The focus for testing will be on the prototype, finding out what the testers like and dislike about the user interface, testing how long a tasks duration is, and determine pain points.

Using Test Driven Development (TDD) process means every time a pull request is made for new code, it is reviewed by other team members, ensuring better quality code. Further, only fully tested and working code gets merged to main branch, this process is used working with microservices, and ensures the final product is readily deliverable. Additionally, using Behaviour Driven Development (BDD) ensures you deliver the right product. A customer’s wants and needs is important and involving the customer in the production process from start to finish means a better result and allows input from the customer for changes during production. Combining TDD and BDD ensures that you deliver a working product that meets the customer’s needs.

Risks

What risks can be identified?

Fortunately, most of the risks identified with project TiMe had a solution. Ultimately, using methods and principles of DevOps has allowed risks to be minimized. However what we did encounter was if users had to create an account, would it deter them from initially using the app; Code intergration breakage; Throttling; JWT attck; and SQL injection. Below explains each of these and the solution to each of the risks.

Having to create an account being a deterrent from using app.

Solution: Allow single use of main feature without having to create an account. However, using without an account will not allow any entered data to be saved.

Code integration breakage

Solution: Developing each function of the project as its own, referred to as Microservices, each tested for vulnerabilities and quality code through an automated test process. Using Test Driven Development (TDD) and Behaviour Driven Development (BDD) to ensure quality end product that works and meets user needs (Rofrano, J. 2022).

Throttling

Cloud Service Providers will provide a certain ‘read limit’. They will hold to agreement, therefore if you exceed the read limit, you receive an error code of “429_TOO_MANY_REQUESTS” (Rofrano, J. 2022) and need to plan how to reduce read instances.

JWT Attack

A poorly implemented JWT verification system leads to various potential attack vectors for JWT’s, one such attack is exploiting flawed JWT Signatures.
Solution: Follow the JWT specification, including the underlying JWS and JWE specifications to obtain a better understanding of how to correctly build this type of authentication system, understand and develop a solution to validate a token correctly so that it is trustworthy and finally not to store sensitive data that can be exploited, such as user flags e.g., isAdmin.

SQL Injection

SQL Injection is the act of maliciously injecting sql commands into input fields to interfere with application logic which then may return sensitive or hidden data, that was never intended to be shared through that mechanism.
Solution: The most simple and effective prevention method is to use parameterized queries rather than concatenation. Parameterized queries do not allow for the structure of a query to be altered through user input. For this to work, the query string must always be a constant, never a case-by-case variable.
